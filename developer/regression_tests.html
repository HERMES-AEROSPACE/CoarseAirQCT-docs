Regression Tests

CoarseAIR ships with a lightweight, deterministic trajectory-level regression suite under test/. The goal is narrow but useful: run a handful of trajectories on every supported collision system with a fixed RNG seed, and compare the resulting trajectory output against a checked-in reference. Any change — byte-for-byte within tolerance — flags a regression. Designed to run in well under 10 minutes on a modern workstation, so it can be invoked after every recompile.

The suite runs automatically after every successful COARSEAIR_INSTALL. The build script in scripts/building/BuildingInstalling.sh invokes test/regression.sh at the end of each install so that any recompile that changes the numerical output is flagged immediately. To skip the post-install run, set COARSEAIR_SKIP_TESTS=1 in the shell before invoking COARSEAIR_INSTALL. The manual command COARSEAIR_TEST [args] is also exposed and forwards any arguments to test/regression.sh.

Layout

test/
├── README.md                    summary and usage
├── regression.sh                main driver (bash)
├── compare.py                   trajectory comparison tool (Python)
├── .gitignore                   ignores test/scratch/
└── systems/
    ├── CHN/
    │   ├── config.sh            per-system overrides
    │   └── reference/
    │       └── trajectories.csv reference output (committed)
    ├── CNO/
    ├── …                        one directory per system
    └── O4/

Each config.sh specifies the production input file to use as a template and the overrides applied at run time:

SOURCE_INPUT_DIR="input/N3/NASA/input"
SOURCE_INPUT_FILE="CoarseAIR.inp"
SEED=42
N_TRAJECTORIES=10
STARTING_LEVEL=1
FINAL_LEVEL=1

The runner copies that input into a scratch directory (test/scratch/<SYSTEM>/) and applies a standardised set of overrides via a Python sed helper:

  • Database Path — pinned to the in-tree dtb/.

  • Generating / Preprocessing / Running / Postprocessing flags — only the trajectory stage runs.

  • Initial Random Nb Seed — pinned to the seed from config.sh (no clock-based seeding).

  • Nb of Trajectories per Level / Bin — pinned to 10.

  • Starting Level / Final Level for molecule 1 — pinned to 1 so exactly one starting level is exercised.

The production input on disk is never mutated.

Usage

From the repository root:

# Run the whole suite:
bash test/regression.sh

# Run only one system:
bash test/regression.sh --only N3

# Run several systems:
bash test/regression.sh --only H3 N3 O3

# Looser tolerance for cross-compiler runs:
bash test/regression.sh --tolerance 1e-5

# Verbose (show CoarseAIR stdout/stderr):
bash test/regression.sh --verbose

Exit code is 0 if every test passes, 1 if any fail.

Generating / refreshing the references

The first time the suite is run, the reference files don’t exist yet and each system reports SKIP  <SYS>  (no reference — run with --update to generate). To populate them, run:

bash test/regression.sh --update

Review the produced test/systems/<SYSTEM>/reference/trajectories.csv files, commit them, and future runs will compare against them.

The same command is used after any intentional numerical change (PES bug fix, sampler improvement, integrator upgrade) — re-run with --update once the build is known-good and commit the refreshed references with a clear note on why the numerics changed.

Tolerance policy

The trajectory file has 10 columns (see Post-processing). The comparator applies:

  • Exact match for the six integer columns (trajectory index, PES index, initial \(v / J\), initial and final arrangement codes).

  • Relative tolerance (default \(10^{-8}\), configurable via --tolerance) for the four float columns (\(b_{\max}\), \(b_i\), \(v_f\), \(J_f\)).

Default \(10^{-8}\) is tight enough to catch real bugs while remaining robust to unavoidable last-bit FP drift on the same compiler. Loosen for cross-compiler comparisons; tighten for bit-for-bit checks.

Disabling the tests

Three mechanisms:

  1. Environment variable. Set COARSEAIR_SKIP_TESTS=1 (in the current shell, export COARSEAIR_SKIP_TESTS=1 in ~/.bashrc, or prepend COARSEAIR_SKIP_TESTS=1 to a single install command). Both the post-install auto-invocation inside COARSEAIR_INSTALL and the standalone bash test/regression.sh driver respect this flag and exit 0 immediately.

  2. Manual invocation only. Call bash test/regression.sh or COARSEAIR_TEST when you want; nothing else triggers the tests.

  3. Per-system skip. To temporarily skip one system without removing its config, delete that system’s reference/trajectories.csv — the runner reports SKIP  <SYS>  (no reference).

Typical output

A successful run looks like:

PASS  test/scratch/CHN/.../trajectories.csv       (10 trajectories)
PASS  test/scratch/CNO/.../trajectories.csv       (10 trajectories)
PASS  test/scratch/CO2/.../trajectories.csv       (10 trajectories)
…
PASS  test/scratch/O4/.../trajectories.csv        (10 trajectories)

=== regression summary ===
  passed: 15
  failed: 0

A regression is flagged on stderr with the column, row, and magnitude of the first discrepancy:

FAIL  test/scratch/O3/.../trajectories.csv
   col 8 (float, rtol=1.0e-08): 3 mismatch(es); first at row 2:
   actual=1.234567890123e+01 expected=1.234567891234e+01
   (abs=1.111e-07, rel=9.000e-09)

What it does and does not catch

Caught by this suite (among others):

  • PES gradient bug changing the trajectory path.

  • Integrator refactoring that shifts timestep selection.

  • Initial-state sampler change that reorders trajectories.

  • Coordinate-transformation bug in the Jacobi setup.

  • Arrangement-classification change.

Not directly caught (need dedicated tests elsewhere):

  • Rate-extraction bugs (post-processing is not exercised).

  • Level-generation bugs (levels are read from the pre-computed database).

  • Cluster/scheduler-related bugs (local runs only).

  • Compiler-induced FP drift beyond the tolerance.

For rate-extraction regression tests specifically, add a follow-up stage that runs coarseair-trajectoriesstats.x and coarseair-posttrajectories.x on the trajectory output and compares the resulting rate files; see the roadmap in Development Roadmap.

Historical note: the VVTC comparison

The previous test/ directory contained CompareToVVTC_N3.m and reference CSVs from David Schwenke’s VVTC code. Those files have been removed — their manual MATLAB-driven comparison has been superseded by the automated multi-system suite described above. Git history preserves them if needed for reference.
